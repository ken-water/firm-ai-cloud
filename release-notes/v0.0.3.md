# CloudOps One v0.0.3

## Release Metadata

- Version: `v0.0.3`
- Release date: `2026-03-02`
- Release type: `minor`
- Git tag: `v0.0.3`

## Executive Summary

`v0.0.3` delivers the first end-to-end security baseline for CloudOps One: OIDC login/session support, bearer-token RBAC compatibility, web-console session UX with role-aware action visibility, and default CI verification for auth/RBAC regression coverage.

## Highlights

- OIDC login callback flow with local user mapping and bearer session issuance.
- Web-console sign-in session model with role-aware read/write experience.
- Security-focused CI and operations documentation for release-grade rollout.

## Added

- OIDC auth APIs:
  - `GET /api/v1/auth/oidc/start`
  - `GET /api/v1/auth/oidc/callback`
  - `GET /api/v1/auth/me`
  - `POST /api/v1/auth/logout`
- OIDC/session persistence schema:
  - `iam_external_identities`
  - `auth_oidc_login_states`
  - `auth_sessions`
- OIDC dev regression smoke script:
  - `scripts/test-oidc-dev.sh`
- GitHub Actions CI workflow for default verification:
  - `.github/workflows/ci.yml`
- Security operations and release-readiness guide:
  - `docs/11-security-operations-v0.0.3.md`

## Changed

- RBAC principal resolution now supports:
  - `x-auth-user` header mode
  - `Authorization: Bearer <session_token>` mode
- Web console now supports:
  - sign-in/sign-out session handling
  - bearer-token bootstrap (`VITE_AUTH_TOKEN`)
  - role-aware action visibility for CMDB/discovery/notification write paths
  - auto sign-in reset when bearer session expires
- Release governance now includes mandatory security checklist coverage for auth/RBAC/audit scope.

## Fixed

- Normalized protected API examples in quickstart to consistently include authentication headers.
- Updated RBAC regression expectation for mixed auth mode error responses.

## Breaking Changes

- None

## Upgrade Steps

1. Pull latest source and deploy database migration set (includes auth session tables).
2. Configure OIDC environment variables if enabling enterprise SSO:
   - `AUTH_OIDC_ENABLED=true`
   - `AUTH_OIDC_REDIRECT_URI=...`
   - additional `AUTH_OIDC_*` endpoints/client settings as required
3. Validate runtime behavior:
   - header mode (`x-auth-user`) for local compatibility
   - bearer mode (`Authorization: Bearer ...`) for OIDC sessions
4. For web-console bearer bootstrap (optional):
   - `VITE_AUTH_TOKEN=<access_token> npm run dev`

## Known Issues

- Full enterprise OIDC hardening (for example PKCE/state nonce policy tuning per IdP) is still baseline-level and may need provider-specific adjustments.
- Workflow request/approval APIs remain reserved in RBAC mapping but are not fully implemented yet.
