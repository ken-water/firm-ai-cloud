# CloudOps One v0.1.15

## Release Metadata

- Version: `v0.1.15`
- Release date: `2026-03-15`
- Release type: `minor`
- Git tag: `v0.1.15`
- GitHub issues: `#193, #194, #195, #196, #197`

## Executive Summary

v0.1.15 closes the next operator gap after readiness repair by turning the remaining notification and ticket follow-up bootstrap steps into a single integration workspace.

This release delivers a deterministic integration catalog, guided bootstrap apply APIs, cockpit inline bootstrap controls, and a release-gate validation script that covers catalog read, apply, verify, idempotency, workspace visibility, and read-only guard behavior.

## Highlights

- Added integration bootstrap catalog for operator notifications and default ticket follow-up policy.
- Added guided apply API with idempotent create-or-enable semantics and audit coverage.
- Added cockpit integration bootstrap workspace for inline review and apply.
- Added v0.1.15 operator validation script with full bootstrap and read-only guard coverage.

## Added

- Planning/release docs:
  - `docs/50-v0.1.15-integration-bootstrap-plan.md`
  - `docs/51-v0.1.15-release-gate-checklist.md`

- v0.1.15 validation suite:
  - `scripts/qa-v0.1.15-operator-journey.sh`

## Changed

- Integration bootstrap catalog now exposes deterministic operator-facing metadata:
  - `recommended_next_key`,
  - per-item `status`,
  - `gap_reason`,
  - `required_inputs`,
  - `default_payload`,
  - `evidence`.
- Bootstrap apply now supports two productized operator tasks:
  - `operator_notifications` bootstraps the runbook-risk notification template, default channel, and subscription,
  - `ticket_followup_policy` enables or updates the default ticket escalation owner.
- Cockpit runbook admin panel now adds:
  - integration workspace visibility,
  - inline suggested inputs,
  - one-click `Apply bootstrap`,
  - immediate refresh of readiness and repair-plan state after apply.
- v0.1.15 scope, dependency order, and release gate remain documented in:
  - `docs/20-v0.1x-operator-simplicity-plan.md`
  - `docs/50-v0.1.15-integration-bootstrap-plan.md`
  - `docs/51-v0.1.15-release-gate-checklist.md`

## Fixed

- Operators no longer need to leave the cockpit flow to translate notification and ticket bootstrap gaps into separate low-level setup actions.
- Re-running the same integration bootstrap request after convergence is now a safe reused/no-op path instead of an ambiguous repeat change.

## Breaking Changes

- None.

## Upgrade Steps

1. Pull `v0.1.15` code and restart API/web services.
2. Ensure `v0.1.14` owner-readiness repair APIs and schema changes are already applied.
3. Verify new/updated APIs:
   - `GET /api/v1/ops/cockpit/integrations/bootstrap`
   - `POST /api/v1/ops/cockpit/integrations/bootstrap/apply`
4. Open the cockpit runbook admin panel and verify the integration bootstrap workspace appears above the owner directory section.
5. Run v0.1.15 operator validation:
   - `bash scripts/qa-v0.1.15-operator-journey.sh --api-base-url http://127.0.0.1:8080 --auth-user admin`
   - `bash scripts/qa-v0.1.15-operator-journey.sh --api-base-url http://127.0.0.1:8080 --auth-user viewer --read-only`

## Known Issues

- v0.1.15 intentionally bootstraps only the two highest-friction operator tasks and does not yet cover broader external-system onboarding.
- Workspace validation remains API-first and does not include browser automation.
