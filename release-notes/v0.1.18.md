# CloudOps One v0.1.18

## Release Metadata

- Version: `v0.1.18`
- Release date: `2026-03-16`
- Release type: `minor`
- Git tag: `v0.1.18`
- GitHub issues: `#211, #212, #213, #214, #215`

## Executive Summary

v0.1.18 turns first-day activation into a repeatable daily-return loop so SMB operators can reopen CloudOps One each day, see what matters now, and record safe progress without leaving the product.

This release adds a daily ops briefing API, auditable follow-up task-state actions, a cockpit workspace for today/overdue/blocked work, and a validation script that proves read, prioritize, act, verify, recommended-action visibility, and read-only guard behavior.

## Highlights

- Added one daily ops briefing that aggregates follow-up work across alerts, tickets, runbook risk, go-live drift, and activation leftovers.
- Added product-owned follow-up actions so operators can acknowledge, complete, or defer tasks with audit context.
- Added a cockpit daily ops workspace focused on today's queue and recommended next action.
- Added a v0.1.18 validation suite that exercises both write-capable and read-only paths.

## Changed

- Overview daily cockpit now adds a dedicated daily ops return-loop workspace with:
  - task summary cards for overdue/blocked/due/completed/deferred counts,
  - recommended next task visibility,
  - inline follow-up actions,
  - recommended-action shortcuts for remediation pages.
- Daily ops briefing now prioritizes follow-up work across:
  - alerts,
  - tickets,
  - repeated runbook failures,
  - go-live readiness drift,
  - activation leftovers and recent activation friction.
- RBAC mapping and protected-route coverage now include the new daily ops read/write contracts.

## Fixed

- SMB operators no longer have to reconstruct the day's work by checking multiple unrelated pages.
- Follow-up completion and defer state is now auditable and product-owned instead of being tracked in external notes or chat.

## Breaking Changes

- None.

## Upgrade Steps

1. Pull `v0.1.18` code and restart API/web services.
2. Ensure the new daily follow-up state migration is applied.
3. Verify the new APIs:
   - `GET /api/v1/ops/cockpit/daily-ops/briefing`
   - `POST /api/v1/ops/cockpit/daily-ops/follow-up-actions`
4. Open the overview cockpit and verify the daily ops return-loop workspace appears above the existing daily queue and checklist sections.
5. Run v0.1.18 operator validation:
   - `bash scripts/qa-v0.1.18-daily-ops-loop.sh --api-base-url http://127.0.0.1:8080 --auth-user admin`
   - `bash scripts/qa-v0.1.18-daily-ops-loop.sh --api-base-url http://127.0.0.1:8080 --auth-user viewer --read-only`

## Known Issues

- v0.1.18 creates a product-owned daily follow-up layer, but it does not yet auto-close source records when operators mark a task complete.
- Daily ops prioritization is now explicit and auditable, but it does not yet include multi-operator ownership balancing or scheduling optimization.
