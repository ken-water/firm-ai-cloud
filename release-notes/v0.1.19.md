# CloudOps One v0.1.19

## Release Metadata

- Version: `v0.1.19`
- Release date: `2026-03-17`
- Release type: `minor`
- Git tag: `v0.1.19`
- GitHub issues: `#217, #218, #219, #220, #221`

## Executive Summary

v0.1.19 upgrades daily operations from a single queue view to an ownership-closure operating loop. Teams can now see explicit owner/due/escalation policy per task, verify carryover at handoff boundaries, and inspect deterministic escalation candidate semantics directly in product APIs and UI.

This release adds policy-backed ownership metadata in daily briefing, a closure continuity API for carryover and escalation triggers, a web workspace update for ownership visibility and closure review, and a dedicated QA script that validates write-capable and read-only owner-closure flows.

## Highlights

- Added explicit owner + due-policy contract on every daily follow-up item.
- Added closure continuity digest API with carryover and escalation candidate semantics.
- Added ownership-focused cockpit UX including closure continuity panel and owner-oriented follow-up path.
- Added v0.1.19 owner-closure validation script with release-gate artifact outputs.

## Changed

- `GET /api/v1/ops/cockpit/daily-ops/briefing` now returns:
  - `owner`,
  - `due_policy`,
  - `due_at`,
  - `escalate_at`.
- Owner derivation chain is now deterministic and visible in evidence metadata:
  - ticket assignee,
  - default ticket escalation owner,
  - runbook-risk configured route fallback.
- Overview daily ops workspace now includes:
  - owner and due-policy visibility in queue rows,
  - closure continuity carryover/escalation panel,
  - owner-oriented follow-up apply action for write-capable users.

## Fixed

- Ownership and closure signals no longer require manual cross-checking across ticket, runbook, and handoff pages.
- Read-only/operator behavior for owner-closure actions is now validated by a dedicated release-gate script.

## Breaking Changes

- None.

## Upgrade Steps

1. Pull `v0.1.19` code and restart API/web services.
2. Ensure API and web-console are running with latest migrations.
3. Verify daily ops endpoints:
   - `GET /api/v1/ops/cockpit/daily-ops/briefing`
   - `GET /api/v1/ops/cockpit/daily-ops/closure-continuity`
   - `POST /api/v1/ops/cockpit/daily-ops/follow-up-actions`
4. Open Overview cockpit and verify:
   - owner/due/escalation fields appear in daily ops rows,
   - closure continuity panel shows carryover and escalation candidates.
5. Run v0.1.19 validation:
   - `bash scripts/qa-v0.1.19-owner-closure-loop.sh --api-base-url http://127.0.0.1:8080 --auth-user admin`
   - `bash scripts/qa-v0.1.19-owner-closure-loop.sh --api-base-url http://127.0.0.1:8080 --auth-user viewer --read-only`

## Known Issues

- Owner follow-up apply in v0.1.19 confirms ownership context in follow-up audit trail; direct owner reassignment workflows remain in ticket/routing configuration surfaces.
- Escalation candidate semantics are deterministic but currently policy-threshold based; adaptive/noise-aware escalation tuning is still future work.
