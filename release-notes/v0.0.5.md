# CloudOps One v0.0.5

## Release Metadata

- Version: `v0.0.5`
- Release date: `2026-03-03`
- Release type: `patch`
- Git tag: `v0.0.5`

## Executive Summary

`v0.0.5` delivers the monitoring bootstrap baseline for private-cloud onboarding and introduces CMDB lifecycle governance needed for production readiness. This release bundles Zabbix server/proxy/agent into one-click deployment, adds monitoring source registry APIs, and enforces lifecycle transition gates through explicit multi-binding requirements.

## Highlights

- Bundled Zabbix stack is now part of one-click install/upgrade and offline delivery, including automatic bootstrap of proxy and local host objects.
- Monitoring source registry APIs were added as the foundation for unified monitoring source management and connection probing.
- CMDB lifecycle gating now requires complete department/business/owner bindings before an asset can become `operational`.

## Added

- Monitoring source registry baseline:
  - migration: `services/api/migrations/202603030001_create_monitoring_sources.sql`
  - APIs:
    - `GET /api/v1/monitoring/sources`
    - `POST /api/v1/monitoring/sources`
    - `POST /api/v1/monitoring/sources/{id}/probe`
  - RBAC permission keys:
    - `monitoring.sources.read`
    - `monitoring.sources.write`
- Monitoring bootstrap implementation plan:
  - `docs/14-v0.0.5-monitoring-bootstrap-plan.md`
- Bundled Zabbix deployment stack in container install:
  - `zabbix-db`
  - `zabbix-server`
  - `zabbix-web`
  - `zabbix-proxy`
  - `zabbix-agent-local`
- Offline package now includes bundled Zabbix images via `cloudops-images.tar`.
- Zabbix bootstrap automation:
  - script: `scripts/bootstrap-zabbix.sh`
  - auto-registers proxy `cloudops-proxy`
  - auto-registers local host `cloudops-local-agent`
  - performs proxy-path local agent availability checks
- CMDB lifecycle and binding baseline:
  - migration: `services/api/migrations/202603030002_create_asset_bindings_and_lifecycle.sql`
  - APIs:
    - `GET /api/v1/cmdb/assets/{id}/bindings`
    - `PUT /api/v1/cmdb/assets/{id}/bindings`
    - `POST /api/v1/cmdb/assets/{id}/lifecycle`
  - multi-binding support for:
    - departments
    - business services
    - owners (`team` / `user` / `group` / `external`)

## Changed

- Install/upgrade workflows now wait for Zabbix services as part of dependency health checks.
- Install/upgrade workflows now invoke `scripts/bootstrap-zabbix.sh` after service readiness.
- Installation and quickstart documents now include:
  - default Zabbix access endpoints and credentials
  - remote agent onboarding configuration guidance
- CMDB asset create behavior now defaults to `idle` status.
- Direct asset creation with `status=operational` is blocked by API validation.
- Discovery-approved asset creation now sets status to `idle` (instead of `active`) to align with lifecycle gate semantics.
- RBAC route mapping and tests now include monitoring source endpoints and permissions.

## Fixed

- Resolved Zabbix first-start schema bootstrap failure on MySQL with binary logging enabled:
  - set `--log-bin-trust-function-creators=1` for `zabbix-db`
  - avoids trigger creation permission failures during initialization

## Breaking Changes

- None

## Upgrade Steps

1. Pull latest `main` and ensure local branch is up to date.
2. Upgrade dependency stack:
   - online mode: `bash scripts/upgrade.sh`
   - air-gapped mode: `bash scripts/upgrade.sh --no-pull`
3. If needed, rerun Zabbix bootstrap explicitly:
   - `bash scripts/bootstrap-zabbix.sh --env-file deploy/.env`
4. Start/verify backend API so new migrations and endpoints are active:
   - `cargo run -p api`
5. Verify web console quality gate:
   - `cd apps/web-console && npm run check:ui`
