# CloudOps One v0.1.14

## Release Metadata

- Version: `v0.1.14`
- Release date: `2026-03-11`
- Release type: `minor`
- Git tag: `v0.1.14`
- GitHub issues: `#187, #188, #189, #190, #191`

## Executive Summary

v0.1.14 closes the next operator gap after owner/readiness visibility by turning readiness findings into concrete repair guidance and one-click bootstrap actions.

This release delivers deterministic repair-plan APIs, idempotent apply APIs for notification bootstrap, cockpit inline repair workflow, and a validation script that covers plan, apply, verify, idempotency, cockpit visibility, and read-only guard behavior.

## Highlights

- Added owner-readiness repair-plan API with deterministic recommended actions.
- Added one-click repair/apply API for template, channel, and subscription bootstrap.
- Added cockpit inline repair workflow inside the existing runbook owner-readiness panel.
- Added v0.1.14 operator validation script with repair and idempotency coverage.

## Changed

- Owner-readiness payloads now map gap states to deterministic repair guidance, including:
  - `action_key`,
  - `resource_kind`,
  - `operation`,
  - `proposed_name`,
  - `proposed_target`,
  - `proposed_channel_type`,
  - `reuse_hint`.
- Owner-readiness repair apply now:
  - creates or enables the runbook-risk notification template when needed,
  - creates or enables the owner notification channel when needed,
  - creates or enables the runbook-risk subscription when needed,
  - returns a safe no-op when readiness is already `ready`.
- Cockpit runbook owner-readiness panel now supports:
  - inline repair-plan visibility,
  - one-click `Apply repair`,
  - immediate refresh of readiness and repair-plan state after apply.
- v0.1.14 scope, dependency order, and release gate remain documented in:
  - `docs/20-v0.1x-operator-simplicity-plan.md`
  - `docs/48-v0.1.14-owner-readiness-repair-plan.md`
  - `docs/49-v0.1.14-release-gate-checklist.md`

## Fixed

- Operators no longer need to infer low-level notification setup steps from readiness error strings alone.
- Re-running readiness repair after the first successful bootstrap no longer produces an unnecessary failure path.

## Breaking Changes

- None.

## Upgrade Steps

1. Pull `v0.1.14` code and restart API/web services.
2. Ensure `v0.1.13` owner-directory/routing schema changes are already applied.
3. Verify new/updated APIs:
   - `GET /api/v1/ops/cockpit/runbook-templates/analytics/owner-readiness`
   - `GET /api/v1/ops/cockpit/runbook-templates/analytics/owner-readiness/repair-plan`
   - `POST /api/v1/ops/cockpit/runbook-templates/analytics/owner-readiness/repair-actions`
4. Open the cockpit runbook panel and verify repair guidance plus one-click repair visibility.
5. Run v0.1.14 operator validation:
   - `bash scripts/qa-v0.1.14-operator-journey.sh --auth-user admin`
   - `bash scripts/qa-v0.1.14-operator-journey.sh --auth-user viewer --read-only`
