# CloudOps One v0.1.20

## Release Metadata

- Version: `v0.1.20`
- Release date: `2026-03-17`
- Release type: `minor`
- Git tag: `v0.1.20`
- GitHub issues: `#223, #224, #225, #226, #227`

## Executive Summary

v0.1.20 completes the daily-ops closed loop from visibility to action. Operators can now reassign task owners inline and execute escalation actions from closure continuity candidates, with deterministic backend guardrails and read-only/write separation preserved in the UI.

This release ships owner assignment and escalation action APIs, cockpit UX updates for inline owner/escalation operations, and a dedicated QA script that validates the read -> owner update -> escalation action -> verify -> read-only guard path.

## Highlights

- Added inline owner assignment path for daily follow-up tasks.
- Added escalation action endpoint with eligibility re-check and audit trail.
- Added closure continuity escalation execution UX for write-capable operators.
- Added v0.1.20 owner/escalation validation script and release-gate artifacts.

## Added

- Planning/release docs:
  - `docs/60-v0.1.20-owner-assignment-escalation-loop-plan.md`
  - `docs/61-v0.1.20-release-gate-checklist.md`

  - `POST /api/v1/ops/cockpit/daily-ops/escalation-actions`
- v0.1.20 validation suite:
  - `scripts/qa-v0.1.20-owner-escalation-loop.sh`

## Changed

- Daily follow-up metadata updates now preserve existing metadata payloads so owner overrides are not dropped by subsequent actions.
- Overview daily ops workspace now includes:
  - owner assignment update action on follow-up rows,
  - escalation action button for closure continuity escalation candidates,
  - explicit read-only hints with write action disable behavior.

## Fixed

- Escalation action apply now re-validates live candidate eligibility server-side to avoid stale-client mutation risk.
- Owner assignment and escalation follow-up no longer require leaving daily workspace.

## Breaking Changes

- None.

## Upgrade Steps

1. Pull `v0.1.20` code and restart API/web services.
2. Ensure API and web-console are running with latest migrations.
3. Verify daily ops endpoints:
   - `GET /api/v1/ops/cockpit/daily-ops/briefing`
   - `GET /api/v1/ops/cockpit/daily-ops/closure-continuity`
   - `POST /api/v1/ops/cockpit/daily-ops/owner-assignments`
   - `POST /api/v1/ops/cockpit/daily-ops/escalation-actions`
4. Run v0.1.20 validation:
   - `bash scripts/qa-v0.1.20-owner-escalation-loop.sh --api-base-url http://127.0.0.1:8080 --auth-user admin`
   - `bash scripts/qa-v0.1.20-owner-escalation-loop.sh --api-base-url http://127.0.0.1:8080 --auth-user viewer --read-only`

## Known Issues

- Escalation apply stage can be skipped by QA script when no eligible escalation candidate exists in current runtime dataset.
- Owner assignment and escalation actions currently use prompt-driven inline input in the web console; structured form UX is planned for a later milestone.
