# CloudOps One v0.1.13

## Release Metadata

- Version: `v0.1.13`
- Release date: `2026-03-11`
- Release type: `minor`
- Git tag: `v0.1.13`
- GitHub issues: `#181, #182, #183, #184, #185`

## Executive Summary

v0.1.13 closes the next operator setup gap after notification evidence and owner routing by making ownership configuration explicit and by showing whether notification coverage is actually ready before teams depend on it.

This release delivers a configurable owner directory, routing rules, readiness gap detection, cockpit setup UX for owner/readiness management, and a validation script that covers configure, route, readiness verification, cockpit visibility, and read-only guard behavior.

## Highlights

- Added configurable owner directory and routing rules for runbook-risk follow-up.
- Added readiness coverage API that reports actionable ownership and notification gaps.
- Added cockpit setup UX for owner directory, routing rules, and readiness visibility.
- Added v0.1.13 operator validation script with configure and readiness verification stages.

## Changed

- `POST /api/v1/ops/cockpit/runbook-templates/analytics/alerts/tickets` now resolves owner routing with:
  - existing assignee,
  - configured owner-routing rule,
  - template/default fallback.
- Owner route metadata now returns:
  - `owner_key`,
  - `owner_label`.
- Cockpit runbook risk setup and alert panels now support:
  - owner directory editing,
  - routing rule editing,
  - readiness state visibility,
  - configured owner-label rendering.
- v0.1.13 scope, dependency order, and release gate remain documented in:
  - `docs/20-v0.1x-operator-simplicity-plan.md`
  - `docs/46-v0.1.13-owner-directory-readiness-plan.md`
  - `docs/47-v0.1.13-release-gate-checklist.md`

## Fixed

- Runbook-risk ownership no longer depends on hardcoded route assumptions.
- Operators can now detect missing notification templates, channels, targets, subscriptions, and disabled owners before automatic follow-up fails silently.

## Breaking Changes

- None.

## Upgrade Steps

1. Pull `v0.1.13` code and restart API/web services.
2. Apply migration `202603110002_create_runbook_risk_owner_routing_tables.sql`.
3. Verify new/updated APIs:
   - `GET /api/v1/ops/cockpit/runbook-templates/analytics/owners`
   - `PUT /api/v1/ops/cockpit/runbook-templates/analytics/owners`
   - `GET /api/v1/ops/cockpit/runbook-templates/analytics/owner-routing-rules`
   - `PUT /api/v1/ops/cockpit/runbook-templates/analytics/owner-routing-rules`
   - `GET /api/v1/ops/cockpit/runbook-templates/analytics/owner-readiness`
   - `POST /api/v1/ops/cockpit/runbook-templates/analytics/alerts/tickets`
4. Open the cockpit runbook panel and verify owner directory, routing rules, readiness table, and owner-label visibility.
5. Run v0.1.13 operator validation:
   - `bash scripts/qa-v0.1.13-operator-journey.sh --auth-user admin`
   - `bash scripts/qa-v0.1.13-operator-journey.sh --auth-user viewer --read-only`
