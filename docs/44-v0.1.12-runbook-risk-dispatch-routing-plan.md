# CloudOps One v0.1.12 Planning (Runbook Risk Dispatch and Ownership Routing)

Date: 2026-03-11  
Version: v0.1  
Target release: `v0.1.12`

## 1. Planning Goal

v0.1.12 closes the next highest-priority operator gap after v0.1.11:
operators can create or reuse a runbook-risk ticket, but they still need the platform to route ownership and notify the right people automatically.

Target outcomes:

1. Runbook-risk ticket follow-up has a deterministic owner route.
2. Ticket create/reuse actions generate notification evidence that operators can verify.
3. Cockpit shows owner and dispatch status without requiring users to inspect raw tickets.
4. Release gate validates the full create -> route -> notify -> verify journey.

## 2. Scope

In scope:

- notification fan-out and delivery evidence for runbook-risk ticket actions,
- deterministic owner routing for runbook-risk tickets,
- cockpit owner/dispatch visibility for runbook risk alerts,
- v0.1.12 validation script and release gate checklist.

Out of scope:

- generic multi-channel notification redesign across unrelated modules,
- team calendar optimization or on-call scheduling engine,
- portfolio-level executive dashboards.

## 6. Risks and Mitigation

| Risk | Impact | Mitigation |
| --- | --- | --- |
| Owner routing is too generic | tickets land with the wrong team and create noise | start with deterministic rules tied to existing ticket/escalation metadata |
| Notification evidence is too opaque | operators still cannot verify who was informed | persist delivery/read model and show it in cockpit |
| Dispatch retries duplicate alerts | users lose trust in automated follow-up | keep dedup-safe dispatch semantics for create/reuse flow |
