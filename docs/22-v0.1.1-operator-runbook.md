# CloudOps One v0.1.1 Operator Runbook (Cockpit + Playbooks + Topology Diagnostics)

Date: 2026-03-05
Version: v0.1.1
Audience: front-line operators with limited terminal experience

## 1. Day-1 Checklist (Initial Setup)

1. Sign in to web console and confirm role at top bar (`admin` or `operator` for execution rights).
2. Open `#/overview` and verify `Daily Operations Cockpit` loads queue items.
3. Open `#/workflow` and verify `Playbook Library` shows built-in catalog.
4. Open `#/topology` and select one relation edge to verify diagnostics panel loads.

Expected result:

- Queue, playbook catalog, and diagnostics all load without API errors.

## 2. Day-2 Triage Flow (Alert -> Action -> Closure)

1. In `#/overview` -> `Daily Operations Cockpit`, pick highest-priority queue item.
2. Use next action:
   - alert item: open alert center or acknowledge
   - ticket item: open ticket or set to `in_progress`
   - sync job item: retry monitoring sync or jump to topology context
3. If remediation is required, go to `#/workflow` -> `Playbook Library`:
   - select playbook
   - fill params
   - run `Dry-Run`
   - for high-risk playbook, copy confirmation token and run `Execute`
4. For topology-related incidents, open `#/topology`:
   - click edge
   - review diagnostics trend/alerts/recent changes
   - follow checklist and quick actions
5. Return to alert/ticket and update status to resolved/closed.

Expected result:

- Every queue item has at least one executable next action.
- High-risk operations always go through dry-run + confirmation.
- Actions remain traceable through execution history and audit logs.

## 3. Guided Demo Sequence (No Terminal)

1. Open `#/overview` and explain queue ranking/rationale fields.
2. Trigger one cockpit write action (for example, acknowledge alert).
3. Open `#/workflow` and run `restart-service-safe`:
   - dry-run
   - execute with confirmation token
4. Open `#/topology`, select edge, show diagnostics checklist and quick actions.
5. Open `#/tickets`, show linked ticket status update as closure evidence.

Demo evidence snapshots:

- Playbook execution history table (`#/workflow`)
- Topology diagnostics panel (`#/topology`)
- Ticket timeline/status (`#/tickets`)

## 4. Rollback / Fallback

If playbook execution fails or confirmation expires:

1. Re-run dry-run to generate a fresh confirmation token.
2. Move incident handling to manual path:
   - open ticket and set explicit owner
   - use topology diagnostics to isolate relation scope
   - run only low-risk playbooks until escalation owner confirms
3. Record exception in ticket note and audit trail.

## 5. Known Limits

- Cockpit ranking is rule-based (not ML) and may require policy tuning by team.
- Diagnostics trend uses monitoring sync/alert context, not full RCA graph inference.
- Browser click-path automation for this runbook is not yet covered by end-to-end UI robot tests.
