# v0.1.5 No-Code Operator Journey Validation Runbook

## Goal

Validate the v0.1.5 operator path end-to-end with one command:

- setup
- alert visibility
- incident command
- escalation dry-run
- handover closure
- backup restore evidence

## One-Command Usage

```bash
bash scripts/qa-v0.1.5-operator-journey.sh
```

Default artifacts:

- `.run/qa/v0.1.5-journey-<RUN_ID>/summary.json`
- `.run/qa/v0.1.5-journey-<RUN_ID>/summary.md`
- stage logs under the same directory (`setup.log`, `alert.log`, ...)

## Read-Only / RBAC-Compatible Mode

For read-only validation without write mutations:

```bash
bash scripts/qa-v0.1.5-operator-journey.sh --read-only --auth-user viewer
```

Write-dependent stages are marked as `skipped` instead of `fail` in this mode.

## Stage Diagnostics

Each stage writes its own log file and the summary includes:

- stage status (`pass` / `fail` / `skipped`),
- compact diagnostics snippet,
- direct log path for failure localization.

## Required Environment

- API reachable (default `http://127.0.0.1:8080`).
- Valid user identity via `x-auth-user` header (`--auth-user`).
- `curl` and `jq` installed.

## Failure Localization Workflow

1. Open `summary.md` and identify failed stage.
2. Open corresponding stage log file.
3. Review HTTP status/body and rerun single scenario manually if needed.
4. Fix config/policy/data and rerun full validation script.

## Expected Release Evidence

Attach to release closure issue:

- `summary.json`
- `summary.md`
- any failed stage logs if non-pass run was observed during hardening

This provides auditable proof that no-code operator workflows are still intact for the target release.
