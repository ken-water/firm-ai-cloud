# CloudOps One v0.1.26 Planning (Role Cockpit + AI Follow-up)

Date: 2026-03-19
Version: v0.1
Target release: `v0.1.26`

## 1. Problem Statement

`v0.1.25` improved demo clarity and feedback capture. Remaining gaps for SMB onboarding-to-value:

- role-specific cockpit focus still depends on manual template selection,
- AI guidance is present but not structured into user-facing intent follow-up loops,
- topology and feedback views are still separated from ownership follow-up context,
- release evidence lacks explicit role-cockpit acceptance checks.

## 2. Goals

1. Provide role-focused cockpit quick scenes for operators/admin/network/business users.
2. Add structured AI follow-up intents that map to deterministic evidence blocks.
3. Strengthen feedback-to-owner follow-up visibility across cockpit and setup.
4. Add role-cockpit acceptance evidence requirements into release closure.

## 3. Scope

- planning and release-gate docs for `v0.1.26`,
- issue-first execution map for role cockpit, AI follow-up, and ownership continuity tracks,
- deterministic UI flow for role scene apply/switch and AI follow-up summaries,
- release validation updates for role-cockpit acceptance artifacts.

## 4. Out of Scope

- full autonomous agent execution,
- external BI/reporting platform integration,
- enterprise SSO provider matrix expansion,
- multi-tenant billing/account portal.

## 5. Strategic Alignment

Aligned with:

- `docs/62-smb-intelligent-cloud-product-strategy-baseline.md`

Primary pillars for `v0.1.26`:

1. faster first-value visibility by role,
2. lower analysis-to-action friction through AI follow-up structuring,
3. clearer ownership continuity for SMB teams,
4. stronger release governance with role-oriented acceptance evidence.

- `#259` to `#264` are closed (or explicitly deferred in release issue),
- role-cockpit scene workflow and AI follow-up cards are user-visible and validated,
- release gates pass before publish:
  - `cargo check --workspace`
  - `cargo test -p api`
  - `cd apps/web-console && npm run check:ui`
  - `make release-publish-dry VERSION=0.1.26`
  - `make release-publish VERSION=0.1.26`
  - `make release-check VERSION=0.1.26`

## 8. Risks and Mitigation

| Risk | Impact | Mitigation |
| --- | --- | --- |
| role scenes add visual complexity | operator confusion | keep scene count small and expose explicit reset/default actions |
| AI follow-up appears generic | low trust | tie follow-up cards to explicit evidence keys and thresholds |
| ownership visibility is partial | missed handoff | always include owner/unassigned indicators in joint views |
| acceptance artifacts drift | release governance gaps | enforce artifact fields in release-note metadata and gate checks |
