# CloudOps One v0.1.x Planning (Operator Simplicity and Efficiency)

Date: 2026-03-09
Version: v0.8
Target track: `v0.1.0` -> `v0.1.1` -> `v0.1.2` -> `v0.1.3` -> `v0.1.4` -> `v0.1.5` -> `v0.1.6` -> `v0.1.7` -> `v0.1.8` -> `v0.1.9`

## 1. Planning Goal

This plan drives CloudOps One toward a practical outcome:

- Small-to-mid enterprises (dozens to thousands of assets)
- Operators can complete daily operations safely with minimal technical background
- Core incident loop is simple, guided, and auditable

## 2. Strategy

Use incremental milestones instead of one big release:

1. `v0.1.0`: build onboarding and alert-loop foundation.
2. `v0.1.1`: add one-click operations and daily guided cockpit.
3. `v0.1.2`: harden enterprise security and scale readiness.
4. `v0.1.3`: operator no-code execution loop + 10k-scale evidence closure.
5. `v0.1.4`: safe-by-default operations governance + long-soak and multi-region reliability evidence.
6. `v0.1.5`: operator autonomy closure with escalation automation, handover, and restore evidence chain.
7. `v0.1.6`: operator simplicity v2 with guided presets, reservation flow, and compliance quality gates.
8. `v0.1.7`: runbook execution closure with explicit policy and guarded live path.
9. `v0.1.8`: runbook standardization and one-click replay to reduce manual errors.
10. `v0.1.9`: runbook analytics and failure hotspot visibility for safer daily triage.

## 5. Definition of Done by Track

Each milestone is done only when:

- All milestone issues are closed.
- `cargo check --workspace`, `cargo test -p api`, and `npm run check:ui` pass.
- Release note/changelog are updated and validated.
- CI run URL(s) and benchmark evidence are linked in release artifacts.

## 6. Risks and Mitigation

| Risk | Impact | Mitigation |
| --- | --- | --- |
| Onboarding still too technical | Low adoption by non-expert users | Force wizard-based golden path and explicit remediation hints |
| Alert noise overload | Operators ignore alerts | Add dedup grouping and policy templates before cockpit rollout |
| Automation misuse | Unsafe operation changes | Use dry-run, confirmation, RBAC, and audit logging |
| Scale regressions at 1k+ assets | Unstable production behavior | Add benchmark profile + SLO gate before v0.1.2 release |
| Identity misconfiguration | Access lockout or privilege drift | Keep break-glass controls and add mapping validation checks |
| Uncontrolled high-risk change windows | Increased outage risk from mistimed actions | Enforce maintenance-window and approval policies before execute/replay |
| Reliability blind spots under long-running or cross-region load | Late regression detection near release | Add long-soak + multi-region benchmark evidence before v0.1.4 closure |
