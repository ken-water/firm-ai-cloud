# CloudOps One v0.1.30 Planning (Scale-out Readiness + External Adoption Guardrails)

Date: 2026-03-20
Version: v0.1
Target release: `v0.1.30`

## 1. Problem Statement

`v0.1.29` established pilot bootstrap and external readiness signals. Before broad external scale-out:

- onboarding quality needs stronger guardrails for repeated self-serve usage,
- rollout status needs tenant/cohort visibility instead of one global view,
- feedback-to-roadmap linkage needs deterministic closure tracking.

## 2. Goals

1. Add scale-out onboarding guardrails for repeated external tenant activation.
2. Add cohort-level rollout board for external expansion tracking.
3. Add structured adoption feedback closure loop with deterministic status.
4. Add scale-out acceptance evidence requirements in release closure.

## 3. Scope

- planning and release-gate docs for `v0.1.30`,
- issue-first execution map for guardrail/rollout/feedback tracks,
- overview workspace enhancements for external scale-out operations,
- release validation updates for scale-out acceptance artifacts.

## 4. Out of Scope

- pricing/commercial packaging engine,
- full tenant billing reconciliation,
- cross-cloud marketplace listing workflows,
- enterprise contract and legal automation tracks.

## 5. Strategic Alignment

Aligned with:

- `docs/62-smb-intelligent-cloud-product-strategy-baseline.md`

Primary pillars for `v0.1.30`:

1. improve repeated self-serve external onboarding success,
2. provide clear cohort rollout progress for scale-out decisions,
3. make feedback-driven product iteration trackable and auditable,
4. enforce scale-out evidence in release governance.

## 7. Definition of Done

- `#283` to `#288` are closed (or explicitly deferred in release issue),
- onboarding guardrails and rollout board are user-visible and validated,
- feedback closure queue supports deterministic ownership/status tracking,
- release gates pass before publish:
  - `cargo check --workspace`
  - `cargo test -p api`
  - `cd apps/web-console && npm run check:ui`
  - `make release-publish-dry VERSION=0.1.30`
  - `make release-publish VERSION=0.1.30`
  - `make release-check VERSION=0.1.30`

## 8. Risks and Mitigation

| Risk | Impact | Mitigation |
| --- | --- | --- |
| guardrails become too strict | onboarding friction | keep defaults safe but overridable with explicit confirmation |
| rollout board noise grows with tenants | low signal clarity | enforce prioritized blocking-first ordering |
| feedback queue lacks closure discipline | roadmap drift | require owner + status + updated_at fields |
| acceptance checks lag implementation | governance regression | enforce metadata/file checks in release script |
