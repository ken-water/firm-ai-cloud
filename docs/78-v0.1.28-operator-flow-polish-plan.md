# CloudOps One v0.1.28 Planning (Operator Flow Polish + Demo Control)

Date: 2026-03-20
Version: v0.1
Target release: `v0.1.28`

## 1. Problem Statement

`v0.1.27` improved visual hierarchy and first-screen readability. Remaining delivery gaps before enterprise demos:

- role actions are faster but still fragmented across modules,
- playlist configuration exists but lacks direct run-state controls for demo operation,
- first-value entry from overview to setup/follow-up still needs clearer shortcuts,
- release evidence should explicitly validate operator flow polish scenarios.

## 2. Goals

1. Add operator flow shortcuts that reduce context switching between overview and setup/follow-up.
2. Add demo control baseline for scene playlist run-state visibility.
3. Improve operational state cues (ready/attention/blocking) across key cockpit blocks.
4. Add operator-flow acceptance evidence requirements into release closure.

## 3. Scope

- planning and release-gate docs for `v0.1.28`,
- issue-first execution map for flow shortcuts, demo control, and state-cue tracks,
- web-console overview/cockpit flow updates,
- release validation updates for operator-flow acceptance artifacts.

## 4. Out of Scope

- full auto-rotation rendering engine for fullscreen kiosk mode,
- backend scheduler for scene playback state persistence,
- mobile-native UX track,
- external reporting integrations.

## 5. Strategic Alignment

Aligned with:

- `docs/62-smb-intelligent-cloud-product-strategy-baseline.md`

Primary pillars for `v0.1.28`:

1. faster operator execution from one entry page,
2. stronger demo control confidence for customer-facing walkthroughs,
3. clearer operational state understanding at first glance,
4. stricter release governance around flow acceptance evidence.

## 7. Definition of Done

- `#271` to `#276` are closed (or explicitly deferred in release issue),
- flow shortcuts and demo controls are user-visible and validated,
- state-cue harmonization is applied to key cockpit panels,
- release gates pass before publish:
  - `cargo check --workspace`
  - `cargo test -p api`
  - `cd apps/web-console && npm run check:ui`
  - `make release-publish-dry VERSION=0.1.28`
  - `make release-publish VERSION=0.1.28`
  - `make release-check VERSION=0.1.28`

## 8. Risks and Mitigation

| Risk | Impact | Mitigation |
| --- | --- | --- |
| shortcut overload increases noise | reduced clarity | keep top shortcuts role-scoped and capped |
| demo control states are ambiguous | operator confusion | use explicit status labels and deterministic defaults |
| state cue updates become inconsistent | trust erosion | reuse one status vocabulary across blocks |
| release evidence misses flow checks | governance regression | enforce artifact fields in release-note metadata and gate checks |
