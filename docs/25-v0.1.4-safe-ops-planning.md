# CloudOps One v0.1.4 Planning (Safe Operations for Non-Technical Teams)

Date: 2026-03-06
Version: v0.1
Target release: `v0.1.4`

## 1. Planning Goal

v0.1.4 focuses on making day-2 operations safer and easier for small-to-mid enterprise teams that do not have dedicated SRE resources.

Target outcomes:

1. High-risk changes are gated by policy (time window + approval), not operator memory.
2. Alert noise is reduced through default dedup/suppression governance.
3. Backup/DR continuity and weekly health review become one-click operator workflows.
4. Release evidence includes long-soak and multi-region stability signals, not only short-run snapshots.

## 2. Scope

In scope:

- Maintenance window and change-freeze guardrails for risky operations.
- Two-person approval lifecycle for high-risk playbooks.
- Alert dedup/suppression governance with explainable outcomes.
- No-code backup policy wizard and scheduled drill orchestration.
- Weekly operations health digest in cockpit.
- `24h` long-soak benchmark profile and drift reporting.
- Multi-region latency/failover simulation baseline.
- `v0.1.4` release closure artifacts and deferred-risk handoff.

Out of scope:

- Full ITSM process engine replacement.
- Automatic CI on push/PR (manual `workflow_dispatch` policy remains).
- Multi-cluster active-active orchestration in product runtime.

## 4. Milestone Plan

### M1: Safe-By-Default Controls

- Target issues: `#119 #120 #121`
- Exit criteria:
  - high-risk execution paths are policy-gated and auditable.
  - operators can understand block reasons and next allowed actions in UI.

### M2: Non-Technical Continuity Operations

- Target issues: `#122 #123`
- Exit criteria:
  - backup/drill policy is configurable from UI without shell commands.
  - weekly digest provides clear risk summary and recommended next actions.

### M3: Release-Grade Stability Evidence

- Target issues: `#124 #125`
- Exit criteria:
  - long-soak and multi-region reports are reproducible and linkable in release artifacts.
  - drift/budget violations are surfaced as actionable outputs.

### M4: Release Closure

- Target issue: `#126`
- Exit criteria:
  - release automation and synchronization checks pass.
  - deferred risks and next-cycle follow-up list are documented.

## 5. Definition of Done (v0.1.4)

- Issues `#118` to `#126` are closed (or explicitly deferred with reason in `#126`).
- Mandatory checks pass:
  - `cargo check --workspace`
  - `cargo test -p api`
  - `cd apps/web-console && npm run check:ui`
- Stability evidence is attached/linked:
  - long-soak summary
  - multi-region summary
  - regression/budget gate outputs
- Release automation gates pass:
  - `make release-publish-dry VERSION=0.1.4`
  - `make release-check VERSION=0.1.4`

## 6. Risks and Mitigation

| Risk | Impact | Mitigation |
| --- | --- | --- |
| Policy rules are too strict and block urgent remediation | Delayed incident response | Add audited override path with explicit reason and expiry |
| Approval flow increases operator friction | Slow execution of valid actions | Limit dual-approval to high-risk playbooks only and surface queue states clearly |
| Suppression policies hide important alerts | Missed incidents | Keep suppression explainable, queryable, and bounded by policy defaults |
| Long-soak runs are expensive | Reduced adoption of evidence path | Keep manual trigger policy and optimize artifact granularity |
| Multi-region simulation mismatches real environment | False confidence | Document assumptions/limits and require environment notes in release evidence |
