# v0.1.17 Release Gate Checklist

Date: 2026-03-16  
Version: v0.1.17

This checklist defines the operator-facing release gate for `v0.1.17`.

## 1. Preconditions

1. API and web-console are running against a migrated database.
2. `v0.1.16` go-live readiness APIs are available.
3. At least one write-capable account and one read-only account are available.
4. Release note metadata includes closed GitHub issue list:
   - `- GitHub issues: \`#205, #206, #207, #208, #209\``
   - release closure issue `#210` is closed after publish evidence is posted.

## 2. One-Command Validation

Write-capable scenario:

```bash
bash scripts/qa-v0.1.17-activation-journey.sh \
  --api-base-url http://127.0.0.1:8080 \
  --auth-user admin
```

Read-only scenario:

```bash
bash scripts/qa-v0.1.17-activation-journey.sh \
  --api-base-url http://127.0.0.1:8080 \
  --auth-user viewer \
  --read-only
```

## 3. Required Stage Coverage

The summary must include these stage keys:

1. `activation_status_read`
2. `starter_template_visibility_read`
3. `activation_progress_apply`
4. `activation_verify`
5. `feedback_capture_apply`
6. `read_only_guard`

## 4. Required Artifacts

Each run must generate:

1. `summary.json`
2. `summary.md`
3. `artifact-index.json`
4. Per-stage logs: `*.log`

Default output directory pattern:

- `.run/qa/v0.1.17-journey-<RUN_ID>/`

## 5. Failure Localization Rule

If `overall=fail`, `summary.md` must contain:

1. failed stage key,
2. endpoint path,
3. remediation pointer,
4. stage diagnostics.

Release cannot proceed until all failed stages are resolved or explicitly deferred in release closure issue.

## 6. Release Gate Commands

Before publishing:

```bash
cargo check --workspace
cargo test -p api
cd apps/web-console && npm run check:ui
```

After release note/changelog finalization:

```bash
bash scripts/release-github-issues-check.sh --version 0.1.17 --notes-file release-notes/v0.1.17.md
make release-publish-dry VERSION=0.1.17
make release-publish VERSION=0.1.17
make release-check VERSION=0.1.17
```
