# CloudOps One v0.1.5 Planning (Operator Autonomy and Execution Continuity)

Date: 2026-03-07
Version: v0.1
Target release: `v0.1.5`

## 1. Planning Goal

v0.1.5 focuses on helping non-technical teams keep incident handling and daily operations continuous across shifts and change windows.

Target outcomes:

1. Incident handling has a guided, auditable command flow instead of chat-only coordination.
2. Escalation and SLA breaches are surfaced automatically before queue collapse.
3. Change operations are visible in a single calendar with conflict warnings.
4. Shift handover preserves unresolved risk context by default.
5. Backup/DR workflows include restore evidence links, not only run status.

## 2. Scope

In scope:

- Incident command panel with owner, ETA, and status transitions.
- SLA-aware escalation policy with queue safeguards.
- Unified change calendar and maintenance conflict checks.
- Shift handover digest with carryover risks and action ownership.
- Restore verification evidence path attached to backup/drill operations.
- Simulation tooling for escalation and failover dry-runs.
- v0.1.5 no-code operator journey validation and release closure.

Out of scope:

- Full ITSM replacement and external ticketing migration.
- Autonomous remediation without operator confirmation.
- Active-active runtime traffic control across regions.

## 4. Milestone Plan

### M1: Operator Command Core

- Target issues: `#128 #129 #130`
- Exit criteria:
  - incident owner/state/ETA is explicit and auditable;
  - escalation risk is surfaced before SLA breach;
  - change-window conflicts are visible before execution.

### M2: Shift Continuity and Resilience Evidence

- Target issues: `#131 #132 #133`
- Exit criteria:
  - handover package preserves unresolved risks and next owner;
  - restore evidence can be attached and reviewed from cockpit;
  - simulation artifacts are reproducible and actionable.

### M3: Validation and Release

- Target issues: `#134 #135`
- Exit criteria:
  - no-code operator journey script validates end-to-end flow;
  - release checks pass and evidence links are attached.

## 5. Definition of Done (v0.1.5)

- Issues `#127` to `#135` are closed (or explicitly deferred with reason in `#135`).
- Mandatory checks pass:
  - `cargo check --workspace`
  - `cargo test -p api`
  - `cd apps/web-console && npm run check:ui`
- Validation evidence is attached:
  - operator journey validation output
  - simulation summary output
  - restore evidence samples (or documented defer reason)
- Release automation gates pass:
  - `make release-publish-dry VERSION=0.1.5`
  - `make release-check VERSION=0.1.5`

## 6. Risks and Mitigation

| Risk | Impact | Mitigation |
| --- | --- | --- |
| Incident command flow adds operator friction | Adoption drop | Keep state model small and action wording explicit |
| Escalation policy creates alert fatigue | Queue noise | Make policy template-driven with severity/SLA thresholds |
| Change calendar misses hidden dependencies | Change collision | Include conflict detection against maintenance/freeze windows |
| Handover digest quality degrades over time | Lost ownership context | Make unresolved-owner/action fields mandatory in carryover rows |
| Restore evidence capture is ignored | False continuity confidence | Require restore evidence link for drill success closure |
