# CloudOps One v0.1.4 Operations Continuity and Weekly Digest Guide

Date: 2026-03-06
Version: v0.1

## 1. Goal

This guide explains how non-technical operators can:

1. Configure backup/DR policy from web console without shell scripts.
2. Run scheduled/manual backup and drill jobs with auditable status.
3. Generate and interpret weekly operations digest in one click.

## 2. Backup/DR No-Code Workflow

UI entry: `#/overview` -> `Daily Cockpit` -> `Backup & DR policy`

API endpoints:

- `GET /api/v1/ops/cockpit/backup/policies`
- `POST /api/v1/ops/cockpit/backup/policies`
- `PATCH /api/v1/ops/cockpit/backup/policies/{id}`
- `POST /api/v1/ops/cockpit/backup/policies/{id}/run`
- `GET /api/v1/ops/cockpit/backup/runs`
- `POST /api/v1/ops/cockpit/backup/scheduler/tick`

Minimum policy fields:

- `frequency`: `daily` or `weekly`
- `retention_days`: integer in `[1, 3650]`
- `destination_type`: `local` / `s3` / `nfs`
- `destination_uri`: validated by destination type prefix
  - `local`: starts with `/` or `file://`
  - `s3`: starts with `s3://`
  - `nfs`: starts with `nfs://`

Drill orchestration fields:

- `drill_enabled`
- `drill_frequency`: `weekly` / `monthly` / `quarterly`
- `drill_time_utc` and optional `drill_weekday`

## 3. Baseline Rollout Steps

1. Open `Backup & DR policy` panel.
2. Configure destination URI and retention days.
3. Save policy and confirm `destination_validated = true`.
4. Trigger `Run backup now` and verify success in `Recent backup/drill runs`.
5. Trigger `Run drill now` and verify success.
6. Run `Run scheduler tick` once to validate schedule path and `next_*` fields.
7. Record the first successful run IDs in weekly digest or release evidence.

## 4. Rollback and Recovery Steps

If rollout causes failures:

1. Keep failed run record for audit (do not delete evidence).
2. Update policy destination and schedule parameters via `Save policy`.
3. Run manual backup again (`run_type=backup`) to verify fix.
4. If drill failures continue, set `drill_enabled=false` temporarily and document reason.
5. Re-enable drill only after one successful manual drill run.

If policy misconfiguration is severe:

1. Revert critical fields with `PATCH /backup/policies/{id}` to last known good values.
2. Keep scheduler tick disabled operationally until manual run passes.

## 5. Weekly Digest One-Click Workflow

UI entry: `#/overview` -> `Daily Cockpit` -> `Weekly operations digest`

API endpoints:

- `GET /api/v1/ops/cockpit/weekly-digest`
- `GET /api/v1/ops/cockpit/weekly-digest/export?format=csv|json`

Digest coverage includes:

- Alert pressure: critical/warning open alerts and suppression thread count.
- Remediation latency: stale open ticket count.
- Approval backlog: workflow + playbook approval queues.
- Continuity risk: backup/drill latest failure counts.
- Auth risk: locked local accounts and local accounts without MFA.

## 6. Operator Interpretation Guidance

Use this quick interpretation order:

1. `open_critical_alerts > 0`:
   - immediate risk; assign owner before handling lower-priority items.
2. `backup_failed_policies > 0` or `drill_failed_policies > 0`:
   - continuity risk; prioritize destination validation + rerun.
3. `workflow_approval_backlog + playbook_approval_backlog > 0`:
   - execution bottleneck; clear approvals to reduce remediation lead time.
4. `suppressed_alert_threads > 0`:
   - governance check required; verify suppression did not hide incidents.
5. `local_accounts_without_mfa > 0`:
   - security debt; schedule MFA enrollment cleanup.

## 7. Audit and Reproducibility Notes

- Digest key format: `weekly-YYYY-MM-DD` (`week_start` based).
- CSV/JSON export content is generated directly from the digest snapshot payload.
- Exported files should be attached to weekly operation review or release closure issue.
- Recommended attachment naming: `<digest_key>.csv` / `<digest_key>.json`.
