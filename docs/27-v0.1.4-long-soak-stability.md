# CloudOps One v0.1.4 Long-Soak Stability Guide

Date: 2026-03-06
Version: v0.1

## 1. Goal

Provide a reproducible `24h` long-soak workflow that non-SRE teams can run before release to catch stability drift that short smoke runs miss.

Deliverables:

1. Periodic benchmark samples with API + SSE snapshots.
2. First-vs-last drift summary for key stability metrics.
3. Policy gate output with pass/fail and actionable metric details.

## 2. Scripts and Artifacts

Primary scripts:

- `scripts/benchmark-long-soak.sh`
- `scripts/benchmark-long-soak-gate.sh`
- `scripts/benchmark-long-soak-policy.json`

Default artifact directory:

- `.run/benchmarks/long-soak-<RUN_ID>/`
  - `samples/sample-xxx/` (each includes profile benchmark artifacts)
  - `drift/summary.json`
  - `drift/summary.md`
  - `gate/summary.json`
  - `gate/summary.md`
  - `summary.json`
  - `summary.md`

## 3. Standard 24h Run

```bash
bash scripts/benchmark-long-soak.sh \
  --profile scale-1k \
  --duration-hours 24 \
  --sample-interval-minutes 240
```

Notes:

- Default behavior runs gate automatically at the end.
- Use `--max-samples` for bounded rehearsal.
- Use `--no-sleep` only for script validation, not release evidence.

## 4. Drift Metrics Covered

Drift summary compares first and last sample on:

- `api_success_rate_avg`
- `api_p95_ms_max`
- `api_p99_ms_max`
- `sse_lag_p95_ms`
- `sse_lag_p99_ms`
- `sse_lag_max_ms`
- `sse_stream_error_count`

Gate policy (`benchmark-long-soak-policy.json`) defines allowed delta budget.

## 5. Triage Path for Gate Failure

When gate result is `FAIL`, use this order:

1. Check `drift/summary.md` and identify first failed metric.
2. If API latency drifts:
   - inspect API benchmark raw artifacts in affected samples;
   - compare endpoint-level `p95/p99` spikes to deployment changes.
3. If SSE lag/error drifts:
   - inspect SSE summary and stream error count per sample;
   - verify stream worker capacity and broker saturation.
4. If success-rate drifts:
   - inspect HTTP status mix in raw API results;
   - correlate with auth/DB/log anomalies in run window.
5. Re-run a shorter confirmation soak (`--duration-hours 4 --sample-interval-minutes 60`) after mitigation.

## 6. Release Evidence Checklist

For v0.1.4 release closure:

1. Attach `summary.md` and `drift/summary.md` to issue `#124` and release issue `#126`.
2. Include exact policy file path and commit SHA in issue comment.
3. If gate fails but release proceeds, record explicit defer rationale in `#126`.
