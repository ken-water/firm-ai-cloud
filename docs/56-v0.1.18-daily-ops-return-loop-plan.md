# CloudOps One v0.1.18 Planning (Daily Ops Return Loop and Follow-Up Closure)

Date: 2026-03-16  
Version: v0.1  
Target release: `v0.1.18`

## 1. Planning Goal

`v0.1.17` proved that a new SMB user can reach first visible value faster.
`v0.1.18` must answer the next product question:
why should that user come back tomorrow and keep operating from CloudOps One instead of falling back to chat, spreadsheets, or memory?

Target outcomes:

1. Operators can open one daily workspace and immediately see what needs attention today.
2. The product turns cross-domain follow-up work into a deterministic, prioritized queue instead of scattered pages.
3. Daily follow-up actions expose safe completion, defer, and evidence-aware status semantics.
4. Release gate validates read -> prioritize -> act -> verify -> read-only guard for the daily loop.

## 2. Scope

In scope:

- daily operations briefing across alerts, tickets, runbook risk, readiness drift, and activation leftovers,
- operator follow-up queue with deterministic priority and due-state semantics,
- one-click safe task-state actions (`acknowledge`, `complete`, `defer`) with audit context,
- cockpit/setup UX for today's work, overdue visibility, and recommended next action,
- `v0.1.18` validation script and release gate checklist.

Out of scope:

- multi-operator scheduling redesign,
- mobile app or push-notification expansion,
- broad workflow engine replacement,
- deep SLA automation beyond the follow-up states needed for the first daily loop.

## 3. Product Strategy

`v0.1.18` links short-term pilot adoption and long-term platform strategy.

Short-term adoption goal:

- create a clear reason for daily return usage,
- reduce operator context-switching after activation,
- show visible progress when follow-up work is completed,
- reveal where pilots get stuck in daily operations, not only in setup.

Long-term platform direction:

- establish CloudOps One as the operator's daily control surface,
- normalize follow-up work across monitoring, ticketing, runbook risk, and setup debt,
- prepare a stronger `v0.2.0` decision based on repeat usage and daily workflow evidence.

## 6. Candidate Daily Ops Coverage

The first daily-return version should stay narrow and useful.
It should cover the recurring tasks that make SMB operators reopen the product each day:

1. overdue alerts or tickets that still need ownership,
2. runbook-risk items that need follow-up,
3. go-live or integration gaps that remain unresolved after activation,
4. due-today checks such as backup/restore evidence or notification readiness repair,
5. a deterministic next action and reason for each task.

Each follow-up item should report:

- `status`: `due_today`, `overdue`, `blocked`, `completed`, or `deferred`,
- `priority`: `critical`, `high`, `medium`, or `low`,
- `summary`,
- `reason`,
- `recommended_action`,
- `due_at` when applicable,
- evidence that explains why the item is in today's queue.

## 8. Risks and Mitigation

| Risk | Impact | Mitigation |
| --- | --- | --- |
| Daily workspace becomes another noisy dashboard | users still leave for external tools | limit the first version to deterministic follow-up items with explicit next action |
| Queue priority is not trusted | operators ignore product ordering | expose reason, due-state, and evidence for every prioritized item |
| State actions drift from source-of-truth behavior | follow-up data becomes misleading | keep actions narrow, auditable, and mapped to existing product-owned records only |
| Product optimizes first-day activation but not repeat usage | pilot retention remains weak | make daily return and task completion a release-scoped success criterion |
