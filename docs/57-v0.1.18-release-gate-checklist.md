# v0.1.18 Release Gate Checklist

Date: 2026-03-16  
Version: v0.1.18

This checklist defines the operator-facing release gate for `v0.1.18`.

## 1. Preconditions

1. API and web-console are running against a migrated database.
2. `v0.1.17` activation APIs are available.
3. At least one write-capable account and one read-only account are available.
4. Release note metadata includes closed GitHub issue list:
   - `- GitHub issues: \`#211, #212, #213, #214, #215\``
   - release closure issue `#216` is closed after publish evidence is posted.

## 2. One-Command Validation

Write-capable scenario:

```bash
bash scripts/qa-v0.1.18-daily-ops-loop.sh \
  --api-base-url http://127.0.0.1:8080 \
  --auth-user admin
```

Read-only scenario:

```bash
bash scripts/qa-v0.1.18-daily-ops-loop.sh \
  --api-base-url http://127.0.0.1:8080 \
  --auth-user viewer \
  --read-only
```

## 3. Required Stage Coverage

The summary must include these stage keys:

1. `daily_briefing_read`
2. `follow_up_queue_read`
3. `follow_up_action_apply`
4. `follow_up_verify`
5. `recommended_next_action_visibility`
6. `read_only_guard`

## 4. Required Artifacts

Each run must generate:

1. `summary.json`
2. `summary.md`
3. `artifact-index.json`
4. Per-stage logs: `*.log`

Default output directory pattern:

- `.run/qa/v0.1.18-daily-loop-<RUN_ID>/`

## 5. Failure Localization Rule

If `overall=fail`, `summary.md` must contain:

1. failed stage key,
2. endpoint path,
3. remediation pointer,
4. stage diagnostics.

Release cannot proceed until all failed stages are resolved or explicitly deferred in release closure issue.

## 6. Release Gate Commands

Before publishing:

```bash
cargo check --workspace
cargo test -p api
cd apps/web-console && npm run check:ui
```

After release note/changelog finalization:

```bash
bash scripts/release-github-issues-check.sh --version 0.1.18 --notes-file release-notes/v0.1.18.md
make release-publish-dry VERSION=0.1.18
make release-publish VERSION=0.1.18
make release-check VERSION=0.1.18
```
