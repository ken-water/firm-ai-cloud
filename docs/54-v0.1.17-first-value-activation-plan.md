# CloudOps One v0.1.17 Planning (First-Value Activation and Pilot Feedback)

Date: 2026-03-16  
Version: v0.1  
Target release: `v0.1.17`

## 1. Planning Goal

`v0.1.17` starts the next product transition after `v0.1.16`:
the platform can now tell operators whether the environment is ready, but it still does not get new SMB users to their first visible value quickly enough, and it does not collect enough structured feedback to guide product decisions from real usage.

Target outcomes:

1. New SMB users can reach a first visible value path within one guided product flow.
2. The product exposes opinionated SMB starter templates instead of expecting users to design everything from scratch.
3. The system records structured activation and friction signals so roadmap decisions can be driven by real pilot behavior.
4. Release gate validates first-value read -> apply -> verify -> feedback visibility -> read-only guard.

## 2. Scope

In scope:

- first-value activation workspace for new deployments,
- SMB scenario starter templates and recommended defaults,
- structured activation telemetry and operator friction feedback capture,
- cockpit/setup UX for activation progress and recommended next step,
- `v0.1.17` validation script and release gate checklist.

Out of scope:

- broad multi-tenant architecture changes,
- large plugin marketplace work,
- browser automation expansion,
- deep vertical-specific customizations beyond reusable SMB starter patterns.

## 3. Product Strategy

`v0.1.17` must serve two goals at the same time.

Short-term feedback loop:

- reduce time-to-first-value,
- lower onboarding hesitation,
- instrument where pilots stall,
- support a fast pilot/customer-feedback cadence.

Long-term platform direction:

- turn repeated SMB onboarding patterns into reusable product templates,
- create a durable data layer for adoption, recommendation quality, and friction analysis,
- prepare the product for a stronger `v0.2.0` decision based on real usage instead of internal intuition.

## 6. Candidate Activation Coverage

The first version should stay focused on the first pilot moments that determine whether an SMB user keeps using the product:

1. activation status summary,
2. recommended starter template,
3. first-value checklist progress,
4. deterministic next action,
5. structured “blocked / confused / not applicable” feedback capture.

Each activation item should report:

- `status`: `ready`, `warning`, or `blocking`,
- `summary`,
- `reason`,
- `recommended_action`,
- optional `feedback_state`,
- evidence that explains why the product made that recommendation.

## 8. Risks and Mitigation

| Risk | Impact | Mitigation |
| --- | --- | --- |
| Activation flow becomes another admin dashboard instead of a short guided path | pilots still fail to reach value quickly | require every blocking state to point to one deterministic next step |
| Template set is too generic to be trusted | users abandon guided setup | keep initial templates narrow and evidence-backed around common SMB environments |
| Feedback capture is noisy or unstructured | roadmap still depends on anecdotes | constrain feedback choices to activation-specific friction categories and store context with the event |
| Short-term pilot asks distort the product | long-term platform coherence degrades | only productize repeated pilot patterns that map to reusable SMB workflows |
