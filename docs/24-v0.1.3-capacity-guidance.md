# CloudOps One v0.1.3 Capacity Guidance (scale-10k)

Date: 2026-03-06
Version: v0.1

## 1. Goal

Provide reproducible `scale-10k` benchmark guidance and map benchmark outputs to practical deployment sizing decisions.

## 2. Profile Definition

`scale-10k` is intended for organizations operating roughly thousands of assets with daily burst updates.

Script defaults:

- API benchmark (`scripts/benchmark-api-load.sh`):
  - `REQUESTS_PER_ENDPOINT=420`
  - `WARMUP_REQUESTS=40`
  - `CONCURRENCY=24`
  - endpoint limits: `cmdb_assets=500`, `monitoring_layer_hardware=320`, `workflow_requests=180`
- SSE benchmark (`scripts/benchmark-sse-burst-smoke.sh`):
  - `BURST_EVENTS=520`
  - `BURST_INTERVAL_MS=50`
  - `STREAM_DURATION_SECONDS=130`

## 3. Fixture Preparation Guidance

Recommended for staging benchmark runs:

1. Start API + dependent stack.
2. Prepare 10k fixture assets in PostgreSQL (or import from pre-generated staging snapshot).
3. Verify at least one active monitoring source and worker processing path are healthy.

Example SQL fixture command (staging only):

```bash
docker compose exec -T postgres psql -U cloudops -d cloudops <<'SQL'
INSERT INTO assets (asset_class, name, hostname, status, site, department, owner, custom_fields)
SELECT
  'server',
  format('scale10k-%s', g),
  format('scale10k-%s.local', g),
  'active',
  CASE WHEN g % 2 = 0 THEN 'dc-a' ELSE 'dc-b' END,
  CASE WHEN g % 3 = 0 THEN 'platform' ELSE 'business' END,
  'benchmark',
  jsonb_build_object('fixture_profile', 'scale-10k', 'fixture_id', g)
FROM generate_series(1, 10000) AS g
ON CONFLICT DO NOTHING;
SQL
```

## 4. Reproducible Benchmark Run

Use committed baseline references:

```bash
bash scripts/benchmark-scale-profiles.sh \
  --profile scale-10k \
  --baseline-api-summary benchmarks/baselines/scale-10k/api-summary.csv \
  --baseline-sse-summary benchmarks/baselines/scale-10k/sse-summary.json

bash scripts/benchmark-regression-guard.sh \
  --profile scale-10k \
  --trend-summary .run/benchmarks/profile-scale-10k-<run-id>/trend-delta/summary.json
```

Expected output artifacts:

- profile summary: `.run/benchmarks/profile-scale-10k-<run-id>/summary.md`
- API summary: `.run/benchmarks/profile-scale-10k-<run-id>/api/summary.md`
- SSE summary: `.run/benchmarks/profile-scale-10k-<run-id>/sse/summary.md`
- threshold gate: `.run/benchmarks/profile-scale-10k-<run-id>/gate/gate-summary.md`
- trend delta: `.run/benchmarks/profile-scale-10k-<run-id>/trend-delta/summary.md`
- regression guard: `.run/benchmarks/regression-scale-10k-<run-id>/summary.md`

## 5. Capacity Interpretation Guide

Use the following heuristic to interpret `scale-10k` outputs for production planning:

1. If gate is `PASS` and regression guard is `PASS`, current deployment class is acceptable for 10k-scale baseline traffic.
2. If API `p95/p99` regressions fail but success rate remains stable, prioritize DB/index/query tuning before adding worker capacity.
3. If SSE lag regressions fail, prioritize worker parallelism, queue depth controls, and stream backpressure checks.
4. If success-rate regression fails, treat as release blocker and freeze rollout until root cause is addressed.

Suggested sizing starting points (single region):

- Small footprint (<= 1k assets):
  - API: 2 vCPU / 4 GiB
  - DB: 2 vCPU / 8 GiB
- Mid footprint (1k-5k assets):
  - API: 4 vCPU / 8 GiB
  - DB: 4 vCPU / 16 GiB
- Large mid-market (5k-10k assets):
  - API: 6-8 vCPU / 12-16 GiB
  - DB: 8 vCPU / 24-32 GiB

These are starting recommendations and must be validated against organization-specific load patterns and retention policy.

## 6. CI Evidence Path

CI benchmark job publishes profile artifacts and enforces trend regression guard.

- Workflow: `.github/workflows/ci.yml` (`benchmark-gate` job, manual trigger)
- Artifact bundle includes profile, gate, trend, and regression summaries.

## 7. Regression Failure Triage

If CI regression guard fails:

1. Open regression summary markdown and review `Impacted Metrics`.
2. Identify whether failure is API latency, API success-rate, or SSE lag/error.
3. Compare with trend delta details to isolate endpoint/metric drift direction.
4. Re-run profile locally with same baseline references to confirm reproducibility.
5. Attach root-cause notes and remediation plan to release issue before retry.

## 8. Known Limitations

- Baseline artifacts are repository snapshots and should be refreshed periodically.
- Single-region results do not model multi-region latency or cross-DC failover behavior.
- Long-soak (24h+) drift analysis is out of scope for this document and should be tracked in v0.1.4 planning.
