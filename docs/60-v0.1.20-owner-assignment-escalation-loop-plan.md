# CloudOps One v0.1.20 Planning (Owner Assignment and Escalation Action Loop)

Date: 2026-03-17
Version: v0.1
Target release: `v0.1.20`

## 1. Problem Statement

`v0.1.19` made owner/due/escalation signals explicit, but operators still lack a direct product-owned path to reassign owner and trigger escalation follow-up from the same daily workspace.

## 2. Goals

1. Operators can assign or update owner per daily follow-up task with audit trail.
2. Escalation candidates can be actioned directly with deterministic guardrails.
3. Web workspace keeps read-only/write split explicit and safe.
4. Validation proves read -> owner update -> escalation action -> verify -> read-only guard.

## 3. Scope

- add owner assignment/update API for daily follow-up tasks,
- add escalation action API for eligible closure-continuity candidates,
- extend web daily ops workspace for inline owner update and escalation action,
- add `v0.1.20` QA script and release gate checklist.

## 4. Out of Scope

- AI-based owner prediction,
- cross-team scheduling optimization,
- external ITSM bidirectional sync.

## 7. Risks and Mitigation

| Risk | Impact | Mitigation |
| --- | --- | --- |
| Owner update becomes non-deterministic | trust erosion | keep explicit source/reason + audit payload |
| Escalation action over-triggers | alert fatigue | guard by candidate eligibility + explicit operator confirmation |
| Read-only guard regresses | unsafe operations | maintain strict RBAC and dedicated guard stage in QA script |
