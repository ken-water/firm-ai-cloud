# CloudOps One v0.1.16 Planning (Go-Live Readiness Workspace)

Date: 2026-03-15  
Version: v0.1  
Target release: `v0.1.16`

## 1. Planning Goal

v0.1.16 closes the next maturity gap after v0.1.15:
the product now exposes more guided setup and bootstrap primitives, but a non-technical operator still cannot quickly answer the most important question before rollout or handover:
"is this environment actually ready to operate safely?"

Target outcomes:

1. Operators can see one explicit go-live readiness status instead of hunting across multiple admin pages.
2. Blocking gaps and warning gaps are separated so teams know what must be fixed first.
3. Each readiness gap maps to a deterministic next action instead of tribal knowledge.
4. Release gate validates read -> act -> verify -> read-only guard for the new workspace.

## 2. Scope

In scope:

- operator-facing go-live readiness snapshot API across core operating domains,
- deterministic remediation metadata for blocking and warning states,
- cockpit/admin go-live readiness workspace,
- v0.1.16 validation script and release gate checklist.

Out of scope:

- deep external vendor onboarding beyond current product-owned integrations,
- browser automation suite expansion,
- broad redesign of every existing setup surface.

## 4. Milestone Plan

### M1: Planning Baseline

- Target issues: `#199`
- Exit criteria:
  - v0.1.16 scope/order is locked in docs;
  - readiness domains and operator value are explicit.

### M2: Readiness Snapshot and Guided Actions

- Target issues: `#200 #201`
- Exit criteria:
  - operators can read a product-level go-live status;
  - blocking domains and suggested next action are deterministic;
  - read-only users can inspect readiness without mutation.

### M3: Operator UX and Release Closure

- Target issues: `#202 #203 #204`
- Exit criteria:
  - cockpit exposes one scanning-friendly go-live workspace;
  - validation artifacts cover read + act + verify + guard;
  - publish/check flow passes.

## 5. Candidate Readiness Domains

Initial domains should stay narrow and product-owned:

1. authentication readiness,
2. monitoring source readiness,
3. operator notification readiness,
4. ticket follow-up readiness,
5. backup/restore readiness,
6. runbook execution readiness.

Each domain should report:

- `status`: `ready`, `warning`, or `blocking`,
- `reason`,
- `recommended_action_key`,
- `recommended_action_label`,
- optional deep-link or bootstrap target.

## 6. Definition of Done (v0.1.16)

- Issues `#199` to `#204` are closed (or explicitly deferred with reason in `#204`).
- Mandatory checks pass:
  - `cargo check --workspace`
  - `cargo test -p api`
  - `cd apps/web-console && npm run check:ui`
- Validation artifacts are attached:
  - `scripts/qa-v0.1.16-operator-journey.sh` outputs and stage logs.
- Release automation gates pass:
  - `make release-publish-dry VERSION=0.1.16`
  - `make release-publish VERSION=0.1.16`
  - `make release-check VERSION=0.1.16`

## 7. Risks and Mitigation

| Risk | Impact | Mitigation |
| --- | --- | --- |
| Readiness score becomes too fuzzy | operators still do not trust the result | keep the first version domain-based and evidence-backed |
| Workspace duplicates existing setup pages without reducing effort | complexity rises instead of falls | require every readiness gap to point to a deterministic next action |
| Scope expands into another large admin rewrite | slow delivery and weak closure | keep v0.1.16 focused on scan-first readiness, not full setup redesign |
