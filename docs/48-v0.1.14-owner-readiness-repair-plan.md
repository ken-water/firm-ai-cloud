# CloudOps One v0.1.14 Planning (Owner Readiness Repair and Bootstrap)

Date: 2026-03-11  
Version: v0.1  
Target release: `v0.1.14`

## 1. Planning Goal

v0.1.14 closes the next operator gap after v0.1.13:
operators can now see owner-routing and notification-readiness gaps, but they still need to understand low-level notification configuration and repair those gaps manually.

Target outcomes:

1. Owner-readiness gaps produce deterministic, operator-readable repair guidance.
2. Operators can apply the minimal repair path from the runbook setup flow without leaving the cockpit context.
3. Notification bootstrap is idempotent so repeated repair attempts stay safe.
4. Release gate validates plan -> apply -> verify -> read-only guard.

## 2. Scope

In scope:

- readiness repair recommendation API for runbook-risk ownership,
- one-click bootstrap/apply actions for missing template/channel/subscription coverage,
- cockpit repair workflow for readiness rows,
- v0.1.14 validation script and release gate checklist.

Out of scope:

- full rotation scheduling or on-call roster design,
- generalized workflow repair for unrelated modules,
- external notification provider redesign beyond current channel/subscription model.

## 6. Risks and Mitigation

| Risk | Impact | Mitigation |
| --- | --- | --- |
| Repair suggestions are ambiguous | operators still need engineering help | return concrete action type, target, and reuse behavior per gap |
| Auto-repair creates duplicate config | notification setup becomes noisy or unsafe | keep create-or-reuse semantics and deterministic naming rules |
| Repair UX hides too much detail | operators cannot trust the change | show what resources will be created or reused before apply |
